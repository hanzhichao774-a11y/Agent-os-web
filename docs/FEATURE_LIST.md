## 功能列表

### F001: AgentOS 大模型驱动的任务执行架构

**ID**: F001
**状态**: Open
**引入版本**: v0.1.0
**类别**: New Feature（架构层 / 跨切面）
**优先级**: High
**创建时间**: 2026-05-06
**最近更新**: 2026-05-06

**原始需求描述**：

AgentOS 是一个由大模型自主驱动的任务执行系统。"对话中由大模型理解、决策、执行"是 AgentOS 的核心架构原则，对应到 WorkAgent 编排（F-013/F-020）、Sub-Agent 动态调用（F-017）、HubAgent Skill 创建引导（F-023）、知识图谱沉淀（F-029/F-030）等多个产品 Feature 的共同前置依赖。

当前实现是「LLM 主路径 + 硬编码降级」的混合模式：意图分类（`classify_intent`）、任务规划（`plan_task`）已经走 LLM，但意图路由的兜底、Skill 的"创建"、实体抽取/排除规则、文档名提取等关键支线全部回落到硬编码关键词列表、字符串模板和正则表达式。这违反了 AgentOS 的核心架构原则，导致系统看起来"由 AI 驱动"，但其行为边界被静态规则锁死。

F001 的本质不是某一个具体功能，而是**贯穿整个任务执行链的架构原则**，它必须先成立，AgentOS 的其它 Feature 才有真正"AI 自主"的语义。

#### 5 条跨切面原则

##### 原则 1：意图理解必须由大模型完成
- 所有用户消息的意图分类、参数抽取、上下文消解都由大模型完成
- 不允许通过预定义的关键词列表（`_*_KEYWORDS`）来判定用户在说什么
- 用户的表达方式不应受限于系统预设的"触发词"

##### 原则 2：任务路由必须由大模型决策
- 用户消息到底走哪个执行路径（直接回答 / 实体抽取 / 实体管理 / 编排 / 创建 Skill），完全由模型决策输出
- 不允许"硬编码触发词降级"——主路径失败时不能回退到关键词匹配，否则等价于硬编码路由始终生效

##### 原则 3：Skill 必须由大模型生成
- 用户说"封装为技能"时，模型应当结合**当前会话已学习的规则、用户偏好、文档特征**生成个性化的 Skill 代码
- 不允许预置 Skill 字符串模板供 `_handle_create_skill` 复制
- 已注册的 Skill 也应支持模型对其逻辑做增量补丁，而不是只能"已存在跳过"

##### 原则 4：规则约束必须由大模型从对话中学习并持久化
- 用户在对话中表达的"不要把 X 抽成实体"、"Y 视为同一类"、"Z 总是排除"等约束，必须由模型解析为结构化规则并持久化
- 后续抽取/管理操作必须自动加载并应用这些规则，避免重复犯同样的错
- 规则要可追溯（可回看"因为哪条规则在哪次对话中被设定"）

##### 原则 5：兜底失败方案必须仍是 LLM
- 模型主调用失败/超时时，兜底方案必须仍由 LLM 完成，例如：
  - 切换到更小/更便宜的模型
  - 简化 prompt 重试
  - 拆分任务后逐步重试
- 不允许出现"LLM 失败 → 关键词匹配"这类回到硬编码的降级链路

**验收标准**：

- [x] 代码库内不再存在意图路由用的硬编码关键词列表（`_ORCHESTRATE_KEYWORDS`、`_DIRECT_ENTITY_KEYWORDS`、`_ENTITY_EXCLUSION_KEYWORDS`、`_CREATE_SKILL_KEYWORDS` 全部移除）— Issue 005 已修复 2026-05-06
- [x] `_keyword_fallback` 类型的关键词降级路径被移除，替换为更小模型/简化 prompt 的 LLM 兜底 — Issue 006 已修复 2026-05-06
- [ ] `_CAPABILITY_TEMPLATES` 字典与 `_handle_create_skill` 的字符串模板复制逻辑被移除
- [ ] Skill 创建调用 LLM 生成完整代码，且生成内容会引用当前已学习的规则
- [ ] 用户在对话中说"不要把章节标题抽成实体"，规则被持久化到可查询的存储中
- [ ] 后续相同类型的实体抽取自动加载该规则，不再产生被排除项
- [ ] `_VALID_INTENTS` 这类硬编码意图枚举支持由 Skill 注册动态扩充，而不是写死 5 类
- [ ] Skill 内部不再嵌套硬编码正则做意图/参数解析（如文档名提取）
- [x] 主调用 LLM 失败时，没有任何执行路径会走回到关键词匹配 — Issue 006 已修复 2026-05-06
