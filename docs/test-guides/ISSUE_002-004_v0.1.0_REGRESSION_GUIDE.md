## 测试场景 A：规则学习意图识别

### A-1：用 rule_learning 意图告知系统规则

**步骤：**
1. 在聊天框中输入：`不要把章节标题抽成实体，记住这个规则`
2. 等待系统响应

**Before（修复前）**：系统不认识这个意图，可能走 direct_answer 直接回答，不保存任何规则

**After（修复后，期望）**：
- 系统响应类似："✅ 已记录规则：**不抽取章节标题作为实体**"
- 规则类型：排除实体类型
- 提示：此规则将在下次实体抽取时自动生效

---

### A-2：多种说法都应被识别为规则学习

以下句式都应触发 `rule_learning` 意图（不需要固定词语）：

| 输入 | 期望响应 |
|------|---------|
| `以后忽略页码，不要把它抽成实体` | "已记录规则：不抽取页码作为实体" |
| `记住：日期不应该是实体` | "已记录规则：不抽取日期作为实体" |
| `下次抽取请跳过文件名` | "已记录规则：不抽取文件名" |

---

### A-3：LLM 解析失败时的友好提示

**步骤：**
1. 输入过于模糊的内容：`balalbala 记住这个`

**期望**：系统返回友好提示（如"无法理解您要设置的规则，请更清晰地描述"），不保存空规则，不报错

## 测试场景 B：规则在抽取时自动生效

### B-1：学习规则 → 重新抽取 → 规则生效

**步骤：**
1. 先执行一次实体抽取（`帮我从文档中抽取实体`），观察抽取结果，记录被抽取的实体类型
2. 输入：`不要把章节标题抽成实体，以后都记住这条规则`，等待确认
3. 再次执行实体抽取（`重新从文档中抽取实体`）
4. 比较前后两次抽取结果

**期望**：
- 第一次抽取：可能包含章节标题类实体
- 第二次抽取：LLM 的 prompt 中已包含"用户学习规则（必须遵守）"文本，规则生效，不再抽取章节标题

**验证方式（查看后端日志）**：  
第二次抽取时，日志中 entity_extractor 构建的 system_prompt 应包含类似：
```
用户学习规则（必须遵守）：
- 排除以下实体类型（模式：章节标题）：不抽取章节标题作为实体
```

---

### B-2：无规则时抽取不受影响

**步骤：**
1. 使用新建项目（没有任何学习规则）
2. 执行实体抽取

**期望**：正常抽取，无报错，排除提示为"（无）"

## 测试场景 C：实体排除自动学习规则（Issue 004）

### C-1：排除操作同时保存规则

**步骤：**
1. 先执行一次实体抽取
2. 输入：`排除 report.pdf 的实体`
3. 观察响应

**Before（修复前）**：
- 系统排除了 report.pdf 的实体
- 下次抽取 report.pdf 时，实体会重新被抽取出来

**After（修复后，期望）**：
- 响应中包含：**"已将来源为「report.pdf」的 N 个实体标记为排除。已同时记录规则：下次从该文档抽取时将自动忽略其实体。"**
- 系统同时保存了 `exclude_source` 规则

---

### C-2：恢复操作不保存规则

**步骤：**
1. 输入：`恢复 report.pdf 的实体`

**期望**：系统恢复实体，但**不保存**任何规则（恢复是可逆操作，不应固化为规则）

---

### C-3：排除后再次抽取，来源实体被自动过滤

**步骤：**
1. 排除 `report.pdf` 的实体
2. 重新执行实体抽取（全量，包括 `report.pdf`）
3. 检查抽取结果

**期望**：LLM 在抽取时已看到规则 `忽略来源 [report.pdf] 的实体`，不会把 `report.pdf` 的内容实体化

## Before / After 对比总结

| 场景 | Before | After |
|------|--------|-------|
| 用户说"不要抽取 X" | 被当作普通聊天回答 | 识别为 rule_learning，保存规则 |
| 下次实体抽取 | 不考虑用户历史反馈 | 自动加载学习规则，注入 LLM prompt |
| 排除操作 | 仅删除当前实体，规则不保留 | 排除 + 保存 exclude_source 规则 |
| 恢复操作 | 仅恢复显示 | 仅恢复显示（正确，不保存规则） |
| 重复抽取同一文档 | 已排除的实体会重新出现 | 规则阻止重新抽取该来源实体 |
