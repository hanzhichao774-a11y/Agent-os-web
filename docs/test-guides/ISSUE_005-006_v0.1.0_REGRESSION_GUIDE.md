# Issue 005 + 006 - 路由层硬编码移除 - 人工回归测试指导

## 功能概述

**Issue ID**: 005 + 006（第 1 波修复）
**版本**: v0.1.0
**测试日期**: 2026-05-06
**测试人员**: [待填写]

**修复内容**：
- 删除了 [`backend/routes/chat.py`](../../backend/routes/chat.py) 中 4 套硬编码意图关键词列表（`_ORCHESTRATE_KEYWORDS`、`_DIRECT_ENTITY_KEYWORDS`、`_ENTITY_EXCLUSION_KEYWORDS`、`_CREATE_SKILL_KEYWORDS`）
- 移除了 `_keyword_fallback` 关键词降级路径
- 重构 `classify_intent` 引入两级 LLM 兜底链（主调用失败 → 简化 prompt 兜底 → 默认 `direct_answer`）

**测试目标**：
1. 验证修复**没破坏**：原本能用关键词触发的意图，LLM 主路径仍能正确分类（正向回归）
2. 验证修复**带来的改进**：用同义但不命中原关键词的说法也能正确路由（这是第 1 波的核心价值）
3. 验证**兜底链行为**：通过日志观察 LLM 失败时不再走关键词

## 测试环境

### 前置条件
- 后端已重启加载新代码
  ```bash
  cd backend
  source .venv/bin/activate
  uvicorn main:app --host 0.0.0.0 --port 8000 --reload
  ```
- 前端已启动：`npm run dev`
- 至少创建一个项目（用于发送消息），并确保知识库中已上传至少 1 个文档（用于实体抽取测试）
- LLM 配置正常（`backend/.env` 中 API Key 有效）

### 观察方式
**重点**：所有测试都需要**同时观察**：
1. **前端响应**：消息是否正确路由（实体抽取/编排/直接回答等）
2. **后端日志**：在启动 `uvicorn` 的终端窗口里查看 `[INTENT] => xxx` 日志

**关键日志变化**：
- 修复前：`[INTENT] LLM 分类 => xxx` 或 `[INTENT] 降级关键词 => xxx`
- 修复后：`[INTENT] => xxx`（不再区分 LLM/关键词，因为关键词路径已删除）
- 兜底成功时：`[INTENT] 主调用失败，已通过简化 LLM 兜底成功分类` + `[INTENT] => xxx`
- 全部失败时：`[INTENT] 主+兜底 LLM 均失败，默认 direct_answer 让 BizAgent 自主响应`

## 边界用例

### BC-001: 空消息
- 输入空字符串，预期前端拦截或后端给出友好提示，不应崩溃
- 后端日志：`[INTENT] => direct_answer`

### BC-002: 极长消息（超过 2000 字）
- 输入一段超长描述，验证 LLM 主路径处理是否正常
- 主调用超时时应触发兜底 LLM（简化 prompt + 缩短超时 5s）
- 后端日志可能出现：`[INTENT] LLM 分类超时(10s, simplified=False)` + `[INTENT] => xxx`

### BC-003: 纯英文消息
- 输入：`Extract entities from the documents in the knowledge base`
- 预期：`[INTENT] => entity_extraction`（旧关键词列表是中文，英文用户在旧实现中容易掉到 `direct_answer`）

### BC-004: 表情符号 / 特殊字符
- 输入：`帮我抽取实体啦 👻🎃`
- 预期：`[INTENT] => entity_extraction`（验证 LLM 不被噪音干扰）
