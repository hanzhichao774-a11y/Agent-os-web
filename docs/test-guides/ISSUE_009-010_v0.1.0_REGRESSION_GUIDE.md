# Issue 009 + 010 - 执行层 LLM 兜底 + Skill 可观测性 - 人工回归测试指导

## 功能概述

**Issue ID**: 009 + 010（Wave 2 修复）
**版本**: v0.1.0
**测试日期**: 2026-05-06
**测试人员**: [待填写]

**修复内容**：

- **Issue 010（执行层 LLM 兜底）**：
  - 新增 [`backend/llm_invoker.py`](../../backend/llm_invoker.py)，提供 `safe_llm_call` / `safe_llm_call_sync` 统一封装
  - 自动重试 3 次（指数退避 1s/2s/4s），可选 fallback agent 二次兜底
  - 关键修复：识别 agno 的"伪成功"（`RunOutput.status == ERROR` 与已知错误字符串前缀如 "Request timed out."）
  - 失败时抛出 `LLMCallError` 让上层精确区分"调用失败"vs"业务结果 0 实体"
  - [`backend/entity_extractor.py`](../../backend/entity_extractor.py) 改用 `safe_llm_call_sync`，并为返回结果加 `status` 三态：`ok`/`failed`/`parse_error`

- **Issue 009（Skill 可观测性 + 缓存 + 失败隔离）**：
  - 重写 [`backend/skills/entity_extract.py`](../../backend/skills/entity_extract.py)
  - 合并 `chat.py` 的全局缓存 + 跨项目实体复制逻辑（已抽过的文件不再调 LLM）
  - 全生命周期日志埋点：`[SKILL:entity_extract] start/cached/extracting/completed/failed/skipped/done`
  - 新增 keyword-only `progress_cb(current, total, label, status)` 回调
  - 单文件失败（status='failed' / 'parse_error' / 异常）不阻塞其他文件，最终消息显式列出失败列表

- **Issue 011 部分（后端 SSE 桥接）**：
  - [`backend/routes/chat.py`](../../backend/routes/chat.py) 实体抽取分支用 `asyncio.Queue` + `loop.call_soon_threadsafe` 把 worker 线程进度事件桥接到 SSE
  - 现在前端可收到 `{"type": "progress", "current", "total", "label", "status"}` 事件（前端渲染待 Wave 3 补全）

**测试目标**：

1. **网络正常**：实体抽取正常完成，进度日志可见，SSE 流出现 progress 事件
2. **网络抖动**（模拟）：safe_llm_call 重试 1-2 次后成功，最终结果正确
3. **网络断开**：safe_llm_call 重试耗尽，单文件标失败，其他文件继续；最终消息精确列出失败/成功
4. **缓存命中**：第二次抽取相同文件时跳过 LLM 调用，日志显示 `cached`
5. **回归保护**：路由层和单元测试无回归（自动化由 pytest 守护）

## 测试环境

### 前置条件

- 后端已重启加载新代码：
  ```bash
  cd backend
  source .venv/bin/activate
  uvicorn main:app --host 0.0.0.0 --port 8000 --reload
  ```
- 前端已启动：`npm run dev`
- 知识库中至少上传 2 个文档（建议命名易识别，如 `test_doc_a.md` / `test_doc_b.md` / `test_doc_c.md`）
- LLM 配置正常（`backend/.env` 中 API Key 有效）
- 浏览器开 F12 → Network → 看 SSE 请求 `chat/stream` 的 EventStream 标签页

### 观察方式

所有测试都需要**同时观察**：

1. **前端响应**：消息能否完成 + 最终消息内容
2. **后端 uvicorn 终端**：完整 `[SKILL:entity_extract]` 日志链
3. **浏览器 EventStream**：是否出现 `data: {"type": "progress", ...}` 事件

## 测试用例

### 场景 A：网络正常 — 完整生命周期日志 + SSE progress 事件

**目的**：验证 Skill 可观测性主路径成立。

**前置准备**：清空数据库实体表（或新建项目避免缓存命中）。

**操作步骤**：

1. 在前端选中或新建一个项目
2. 确保知识库已上传 ≥ 2 个文档
3. 在对话中发送：`帮我从知识库里抽取实体`

**通过判定**：

| 检查项 | 期望 | 实际 |
|--------|------|------|
| 后端日志包含 `[SKILL:entity_extract] start: project=... files=N` | ✅ | [ ] |
| 后端日志为每个文档输出 `[SKILL:entity_extract] extracting <name> (... chars)` | ✅ | [ ] |
| 后端日志为每个抽取成功的文档输出 `[SKILL:entity_extract] completed <name>: N entities, M relations` | ✅ | [ ] |
| 后端日志结尾有 `[SKILL:entity_extract] done: N ok / 0 cached / 0 failed` | ✅ | [ ] |
| 浏览器 EventStream 出现 `{"type": "progress", "current": 1, "total": N, "label": "<file>", "status": "extracting"}` | ✅ | [ ] |
| 浏览器 EventStream 出现 `{"type": "progress", ..., "status": "completed"}` | ✅ | [ ] |
| 前端最终消息包含 `已从 N 个文档中提取 X 个实体和 Y 条关系` | ✅ | [ ] |
| 切换到「图谱」标签页，能看到抽取的实体节点 | ✅ | [ ] |

| 浏览器 EventStream 的 progress 事件 status 为 `cached` | ✅ | [ ] |
| 最终消息包含 `（其中 N 个文档命中缓存，未重复调用 LLM）` | ✅ | [ ] |

### 场景 D：网络断开 — 重试耗尽 + 失败隔离

**目的**：验证 Issue 010 的重试机制 + Issue 009 的失败隔离机制。

**前置准备**（任选一种方式模拟）：

- 方式 1（最干净）：在 `.env` 中暂时把 `OPENAI_API_KEY` 改成无效值（如 `sk-invalid-test-12345`），重启后端
- 方式 2：用 `tc` / `pf` 屏蔽 OpenAI 域名（高级用户）

**操作步骤**：

1. 知识库至少 3 个文档；清空数据库实体表（或用全新项目）
2. 发送：`帮我从知识库里抽取实体`
3. 等待完整流程（每个文档应触发 3 次重试 × 退避 1s+2s+4s ≈ 7 秒后失败，3 个文档 ≈ 21 秒）

**通过判定**：

| 检查项 | 期望 | 实际 |
|--------|------|------|
| 后端日志包含至少一条 `[ENTITY] LLM 调用失败 (attempts=3): ...` | ✅ | [ ] |
| 后端日志为每个文档输出 `[SKILL:entity_extract] failed <name>: ...` | ✅ | [ ] |
| **失败的文件不会阻塞下一个文件继续尝试**（连续看到多个 failed 而非中断） | ✅ | [ ] |
| 最终消息显式包含 `以下文档 LLM 调用失败：<file_a>（...）, <file_b>（...）` | ✅ | [ ] |
| 最终消息**不**伪装成 `已从 N 个文档中提取 0 个实体和 0 条关系`（这是修复前的旧 bug） | ✅ | [ ] |
| 浏览器 EventStream 的 progress 事件含 `status: "failed"` 条目 | ✅ | [ ] |
| 测试完成后立即恢复 `.env` 中的有效 `OPENAI_API_KEY` | ✅ | [ ] |

### 场景 E：网络抖动 — 重试后成功（可选 / 较难复现）

**目的**：验证重试机制在间歇性失败下能恢复成功。

**复现方式（较难）**：

- 启动一个本地代理（如 mitmproxy 脚本），让它对 `api.openai.com` 的前 1-2 个请求注入 503，第 3 个请求放行
- 或者改 `entity_extractor.py` 临时把 `max_retries` 设为 5 + 在 `safe_llm_call_sync` 里手工 monkeypatch fake agent，但这等价于单元测试已覆盖

**通过判定**：

| 检查项 | 期望 |
|--------|------|
| 同一文件后端日志可见多次 OpenAI 请求（重试） | ✅ |
| 最终该文件 `[SKILL:entity_extract] completed <name>` | ✅ |
| 单元测试 `test_safe_llm_call_retries_on_exception_then_succeeds` 已自动守护此场景 | ✅ |

> **建议**：场景 E 优先依赖 `pytest tests/test_llm_invoker.py -v` 的自动化守护，手动复现成本过高。

## 已知局限（待 Wave 3 补全）

1. **前端尚未渲染 progress 事件**：浏览器 EventStream 能看到事件，但 UI 仍是干转圈（Issue 011 前端部分）
2. **fallback agent 暂未启用**：当前 entity_extractor 只用主 agent 重试 3 次，未配置 fallback；后续 Wave 可在 LLM 配置中加入备用模型
3. **并发抽取尚未引入**：Skill 仍是串行处理文件（避免 SSE 桥接 + 多线程 progress_cb 竞争）；Wave 3.5 + Issue 011 一起处理
4. **`_run_entity_extraction_sync`（chat.py 内联版）暂保留**：标记 DEPRECATED 但未删除；待 Wave 3 合并为单一入口
